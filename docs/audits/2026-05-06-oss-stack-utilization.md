# OSS Stack Utilization Audit — 2026-05-06

Read-only audit of all OSS components in the Glad Labs / Poindexter stack. Generated by background agent ab45e21d69a25aa1b.

## TL;DR

- **Tracing is the biggest gap**: Tempo deployed, only 3 production code paths emit spans (`services/llm_providers/dispatcher.py`, `plugins/llm_resilience.py`, `brain/health_probes.py`). The dispatcher has **zero hot-path callers** — the writer pipeline still calls `OllamaClient` directly via `services/ai_content_generator.py` (admitted in `migrations/0160_litellm_cutover_default_providers.py:55-69`). `enable_tracing` defaults to `false` (`services/settings_defaults.py:290`).
- **Langfuse is the closest to load-bearing**: prompt management is wired (`services/prompt_manager.py:317-391`); `litellm.success_callback = ["langfuse_otel"]` registered at lifespan startup (`main.py:275-289`). But because the dispatcher has zero callers, span volume is ~zero today. Wiring problem, not Langfuse problem.
- **Prefect is dormant**: One flow exists (`services/pipeline_flow.py`, 463 lines); `task_executor.py:538-541` still imports the legacy `process_content_generation_task`. Five `@task` chunks defined; zero production runs.
- **GlitchTip / Pyroscope / Uptime Kuma**: present but lightly used. GlitchTip triage probe runs (`brain/glitchtip_triage_probe.py`, 720 lines), but worker `sentry_dsn` never seeded → no errors captured. Pyroscope agent Linux-only (`pyproject.toml:101`), default off, no Windows wheel. Uptime Kuma container up, zero monitors provisioned.
- **Recommended consolidation, NOT decommission**: Make Langfuse the unified LLM-trace surface and Tempo the unified service-trace surface, correlate via OTel context. Highest-leverage move: get `LiteLLMProvider` on the writer hot path (one migration) — closes the loop on Langfuse traces, cost tracking, provider observability simultaneously.

**Recommendation — Langfuse vs Tempo split**: Keep both, with explicit roles. Langfuse = LLM-call substrate (prompts, per-prompt cost, latency, model attribution, UI Matt opens). Tempo = service-graph substrate (FastAPI request → DB query → LLM call → R2 upload, all under one trace_id). The `langfuse_otel` callback already emits OTLP to `LANGFUSE_HOST/api/public/otel`, so a single OTLP collector sidecar (or a `tempo` `remote_write` config tweak) can fan-out the same span batch to both backends. Net: Matt opens Langfuse for "why did this draft cost $0.03?" and Grafana/Tempo for "why was this request 12s?". Pyroscope correlates via OTel exemplars once `enable_tracing` is on.

## Component-by-component scorecard

### Prefect 3 — DORMANT (~5%)

- Containers up: prefect-server, prefect-services, prefect-redis (`docker-compose.local.yml:415-479`).
- One `@flow` (`content_generation_flow`), five `@task` decorators (`pipeline_flow.py:71-185`).
- Production callers: zero. `task_executor.py:538-541` imports the legacy entry point.
- Verdict: **wire on the hot path** — one-line import swap.

### Langfuse v3 — WIRED for prompts, DORMANT for traces (~15%)

- Containers up: web/worker/clickhouse/minio/redis (`docker-compose.local.yml:497-647`).
- Prompts wired (`prompt_manager.py:317-391`). LiteLLM callback registered.
- Gap: callback only fires when something calls `litellm.acompletion`. Dispatcher has zero hot-path callers.
- Verdict: **highest-leverage component to fix**. Ship migration + dispatcher routing → 0% → 100% LLM trace coverage.

### LangGraph — WIRED, partial (~35%)

- `services/template_runner.py` (978 lines) is a real `StateGraph` with `AsyncPostgresSaver` checkpointer (default off, `settings_defaults.py:296`).
- One production user: dev_diary template (`run_dev_diary_post.py:300`).
- Verdict: legitimate co-existence; flip checkpointer default to true after smoke test.

### Pyroscope — LIVE (self-profiling), APP-PROFILES NOT WIRED (~10%)

- Container running, profile blocks rolling every ~5min (`docker logs poindexter-pyroscope` shows `process_cpu` head writes).
- BUT only Pyroscope's own Go runtime is being profiled — no app (worker/brain/voice) ships profiles to it. `services/profiling.py:setup_pyroscope` is fully implemented but never called from `main.py`/brain startup. `Dockerfile.worker` doesn't install the `[profiling]` extras either, so the SDK isn't even in the image.
- `pyproject.toml:101` Linux-only marker only blocks installs on Matt's Windows HOST — the worker/brain Linux containers can install fine. **Linux migration is NOT a prerequisite.**
- Verdict: live and producing data, but app-level rollout pending. Tracked: Glad-Labs/poindexter#406.

### GlitchTip — WIRED inbound, DORMANT outbound

- Triage probe (`brain/glitchtip_triage_probe.py`, 720 lines) actively reading GlitchTip API.
- `sentry_dsn` never seeded → worker `sentry_sdk.init()` is a no-op.
- Verdict: half-wired. Either seed the DSN or document the asymmetry.

### Alertmanager — WIRED, real load-bearing

- Routes everything to `worker:8002/api/webhooks/alertmanager`. Brain `alert_dispatcher.py` (891 lines) fans out to Telegram/Discord.
- Verdict: most thoughtfully wired piece in the stack. No changes.

### Uptime Kuma — DEPLOYED, NEVER PROVISIONED
