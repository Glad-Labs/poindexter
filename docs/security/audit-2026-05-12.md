# Security Audit — 2026-05-12

Three parallel auditors (auth+secrets, supply chain+deps, code/CI/exposure)
read the actual code (not docs) and reported back. This is the synthesized
punch list. All sensitive values referenced in findings have been redacted
here; see the per-secret rotation runbook below.

## Trigger

PyPI quarantined the `guardrails-ai` package on 2026-05-12 after a suspect
0.10.1 release. The quarantine blocked CI on every PR. Investigation
confirmed our installed 0.7.3 wheel was uncompromised (hash matched
poetry.lock), but the incident prompted a broader audit — Matt asked
"what else looks like this?"

## P0 — Active leaks / fix today

### 1. Live secrets shipped to the public OSS mirror in `0000_baseline.seeds.sql`

**Three rows had real production values committed with `is_secret='f'`:**

- `discord_ops_webhook_url` — Discord webhook URL with embedded bearer
  token. Anyone reading the public mirror can post to the operator's
  ops channel.
- `indexnow_key` — IndexNow site-ownership key. Lets a third party
  submit URLs to the operator's IndexNow account.
- `langfuse_public_key` — identifies the operator's Langfuse project.
  Public key alone can't authenticate writes, but it's operator-
  identifying metadata that doesn't belong on the OSS mirror.

**Operator actions:** rotate all three upstream (regenerate the Discord
webhook in Server Settings → Integrations → Webhooks; generate a new
IndexNow key; re-issue the Langfuse public key project-side), then
`poindexter settings set <key> <new>` to reseat.

**Code actions (this PR):** seed values replaced with `''`, `is_secret`
flipped to `'t'`. Migration `20260512_032900_redact_leaked_app_settings_secrets.py`
runs the same UPDATE on existing installs.

### 2. `/voice/join` mints 7-day LiveKit tokens with no auth

Route was mounted on the worker FastAPI, exposed over Tailscale **Funnel**
(public internet, not tailnet-only as the docstring claimed). Any visitor
to `https://<funnel-host>/voice/join` got a 7-day `roomJoin/canPublish/
canSubscribe/canPublishData` JWT for Matt's voice room. The route also
fell back to a hardcoded dev secret (`devsecret_change_me_change_me_change_me`)
if `LIVEKIT_API_SECRET` was unset.

**Fix (this PR):** route gated behind `Depends(verify_api_token)`; the
dev-secret fallback now raises `503` instead of minting forgeable tokens.

### 3. Open Dynamic Client Registration (DCR) + no per-route scope enforcement

The OAuth provider at `/register` (RFC 7591) was enabled with no auth
required and `default_scopes=["mcp:read", "mcp:write"]`. Combined with
`verify_api_token` checking only signature/expiry/issuer (no scope
enforcement), any unauthenticated visitor could:

1. `POST /register` → mint a fresh `(client_id, client_secret)` pair
2. `POST /token` with `client_credentials` grant → get a JWT
3. Hit every authenticated route — `/api/cms/*`, `/api/settings/*`,
   `/api/export/rebuild`, etc.

This collapsed the entire OAuth boundary to "anyone with the public
endpoint URL is Matt."

**Fix (this PR):** DCR gated behind `app_settings.oauth_dcr_enabled`
(default `false`). Operators flip the flag briefly when onboarding a
new consumer, register the client, capture the credential, flip back.

**Follow-up (separate PR):** per-route scope enforcement via
`Depends(require_scope("api:write"))`. Today every JWT-holder can hit
every route; that needs to be reduced.

### 4. Unauthenticated routes leak audit log + memory data — RESOLVED 2026-05-12

- `GET /api/pipeline/events` and `/api/pipeline/events/task/{id}` —
  no auth dependency. Returns `audit_log` rows.
- `GET /api/memory/stats`, `/api/memory/search`, and the HTML dashboards
  at `/pipeline` + `/memory` — no auth. The search accepts free-text
  vector queries and returns matched memory rows.

**Fix:** `Depends(verify_api_token)` added to all six endpoints
(`routes/pipeline_events_routes.py`, `routes/memory_dashboard_routes.py`).
Regression tests in
`tests/unit/routes/test_pipeline_events_and_memory_auth.py` confirm
unauthenticated requests return 401 — pinning the contract so a future
refactor can't silently un-gate them.

### 5. SSRF in URL-seed scraper

`POST /api/topics/seed-url` is auth-required, but `services/url_scraper.py`
accepts any `http(s)://` URL and follows redirects with no IP filter.
An authenticated operator (or anyone who can mint a token per finding
#3) can drive scraping against `http://127.0.0.1:9091` (Prometheus),
`http://localhost:18443` (pgAdmin), `http://169.254.169.254/` (cloud
metadata if ever cloud-hosted), `http://<tailnet-ip>:<any>` (any
tailnet service). Title/content of internal pages are exfiltrated
through the task metadata response.

**Follow-up (separate PR):** resolve hostname to IP before request,
reject RFC1918 / 127/8 / 169.254/16 / IPv6 link-local + ULA + ::1.
Re-resolve after each redirect.

**Resolved 2026-05-12** in `services/url_scraper.py` (guard logic at
`_is_blocked_ip` / `_resolve_and_check` / `_safe_get`, lines ~52-269).
Initial request and every redirect re-resolve the hostname via
`socket.getaddrinfo` and reject loopback / RFC 1918 / link-local +
cloud-metadata / Tailscale CGNAT / IPv6 ULA + link-local. Operator
override: `app_settings.url_scraper_allow_internal_ips` (default
`false`). Test coverage: `tests/unit/services/test_url_scraper_ssrf.py`
(54 cases — IP literal denylist, DNS resolution paths, redirect
chains, override flag). Threat model + denylist + override knob +
test matrix documented in
[`docs/architecture/url-scraper-ssrf-guard.md`](../architecture/url-scraper-ssrf-guard.md).

### 6. XSS in `/preview/{token}` — reflects LLM-controlled HTML

The preview page embeds `title`, `excerpt`, and full markdown `content`
**unescaped** into the response HTML. Content originates from research
sources + LLM output, both of which can be attacker-influenced (per
finding #5 SSRF, and prompt-injection-via-research per the code-injection
agent's F11). A poisoned research page → LLM writes `<script>...</script>`
into the body → operator opens the preview link from Telegram on their
phone → script runs on the Funnel-exposed origin.

**Follow-up (separate PR):** render markdown via a sanitizer
(bleach / markdown-it with HTML disabled), `html.escape` title/excerpt,
emit a strict `Content-Security-Policy` header on preview responses.

## P1 — This week

### 7. Alertmanager webhook static-Bearer fallback still in code

`alertmanager_webhook_routes.py` falls back to `hmac.compare_digest`
against `app_settings.alertmanager_webhook_token` when JWT verification
fails. Documented as an exception in the file header — but the claim
elsewhere that #249 removed "the static-Bearer fallback globally" is

via `mint-grafana-token`, remove the static-Bearer branch.

### 8. Secrets read with `site_config.get()` instead of `get_secret()`

Six call sites violate the "secrets must use async `get_secret`" rule:
`sentry_integration.py` (sentry_dsn), `llm_providers/litellm_provider.py`

- `langfuse_experiments.py` + `prompt_manager.py` (langfuse_public_key
- langfuse_secret_key), `publish_service.py` (indexnow_key),
  `task_executor.py` + `morning_brief.py` + `webhook_delivery_service.py`
  (discord_ops_webhook_url + openclaw_webhook_url).

These will silently break if anyone promotes the key to `is_secret=true`
because `site_config.get()` returns `""` for cached-out secret rows.
The 2026-05-12 hotfix promoted three of these to `is_secret=true` →
their call sites must migrate to `get_secret` in the same PR or the
features break silently. (This PR's migration handles the rows; the
follow-up handles the readers.)

### 9. JWT revocation is incomplete

`oauth_provider.revoke_token` marks the _client_ revoked but not the
token. `verify_token` doesn't consult `oauth_clients.revoked_at` or a
`jti` denylist. Compromised client_secret rotation leaves a 60-min
hot window where stolen tokens still verify.

**Action:** add a `jti` denylist table; check on every verify. Or
shorten access-token TTL to 5 min and require re-mint.

### 10. `dev-token` bypass active in non-production envs

`middleware/api_token_auth.py` accepts the literal `"dev-token"` when
`app_settings.development_mode='true'` AND `ENVIRONMENT != "production"`.
The production check uses `os.getenv("ENVIRONMENT", "")` — unset means
"not production" → bypass works. Belt-and-braces: also require
`app_settings.environment != 'production'` in the bypass check, OR
delete the bypass and use a CLI-minted dev OAuth client.

### 11. `POINDEXTER_SECRET_KEY` not generated by `poindexter setup`

`_generate_secrets()` returns 13 keys; `poindexter_secret_key` (the
master key for JWT signing + app_settings envelope encryption) is not
one of them. Operators who don't manually edit bootstrap.toml after
`poindexter setup` will hit a fail-loud on first worker boot
(intentional — better than silent default). But the docs claim it's
auto-generated. **Action:** add `poindexter_secret_key = secrets.token_urlsafe(48)`
to `_generate_secrets`; document the rotation procedure.

### 12. No structured-log secret redaction — **RESOLVED**

`services/logger_config.py` configures structlog with no redaction
processor. Any caller doing `logger.info("loaded", token=tok)` would
write the secret to Loki + disk. Sampled callers don't currently leak,
but the gap is one wrong logger call away from a bad incident.

**Resolution:** structlog + stdlib secret-redaction processor landed
in `services/logger_config.py` (`redact_secrets` at L217 +
`SecretRedactionFilter` at L251). The processor sits between the
timestamper and the renderer in `configure_structlog`'s processor chain
(`services/logger_config.py:340`, hook at L367); the stdlib filter is
attached to every handler in `configure_standard_logging`. Tests at
`tests/unit/services/test_logger_redaction.py` (30 cases) pin every
keyword + value-shape rule below.

**Redaction rules:**

1. **Key-name redaction (case-insensitive).** Any event_dict key whose
   name matches this regex has its value replaced with
   `***REDACTED***`:

   ```text
   (?i)(token|secret|password|api_key|api-key|authorization|cookie|
        bearer|dsn|signing_key|signing-key|access_key|access-key|
        client_secret|client-secret|webhook_url|webhook-url|
        x[-_]revalidate[-_]secret|x[-_]api[-_]key|langfuse_secret|
        discord_ops_webhook|indexnow_key)
   ```

   `re.search` semantics — suffixed/prefixed variants like
   `discord_ops_webhook_url`, `x_api_key_header`, and `sentry_dsn` all
   match.

2. **Value-shape redaction.** Any _string value_ (regardless of key
   name) whose shape obviously screams "credential" is also masked:
   - prefix `Bearer ` / `bearer ` (HTTP Authorization header)
   - prefix `pk-lf-` (Langfuse public key)
   - prefix `sk-lf-` (Langfuse secret key)
   - prefix `pdx_` (Poindexter-issued tokens)
   - prefix `enc:v1:` (envelope-encrypted sentinel from `plugins/secrets.py`)
   - prefix `sk-` (OpenAI-style API keys)
   - prefix `ghp_` / `gho_` (GitHub PAT / OAuth)
   - prefix `xoxb-` / `xoxp-` (Slack bot / user tokens)
   - regex `^Basic [A-Za-z0-9+/=]{8,}$` (HTTP Basic auth)

   This catches the "loaded auth header into a generic `data` field"
   pattern that key-name rules can't see.

3. **Recursion.** Walks dicts AND lists, capped at depth 5 to prevent
   runaway recursion on circular references. Beyond the cap the
   subtree is returned unmodified — degraded redaction is better than
   a logger crash.

4. **Defensive.** Any exception inside the redactor is swallowed (one
   warning line to stderr) and the original event_dict is returned.
   The stdlib filter always returns `True` so log records never get
   dropped on a redactor bug.

**How to add a new secret key class.** Open
`services/logger_config.py` and extend `SECRET_KEY_PATTERN`
(L131) with another alternative — the regex is one giant
case-insensitive `|`-list, so adding a new key family is one line.
Add a matching test case to `tests/unit/services/test_logger_redaction.py`
to lock in the behaviour. For a new credential _value shape_ (a new
prefix or a new regex pattern), extend `SECRET_VALUE_PREFIXES`
(L143) or `SECRET_VALUE_PATTERN` (L160) instead — same
file, same test file, same one-line change.

## Automated key-rotation design

**Goal:** every secret has a documented rotation cost ≤ 60 seconds of
operator wall-clock. The two-step is:

1. **Rotate upstream** — generate the new value at the source (Discord
   UI, IndexNow site, etc.). Operator-driven; can't be automated.
2. **Reseat locally** — write the new value into our config plane.
   Automatable via `poindexter auth rotate <name>` CLI.

### Plumbing

- **DB-backed secrets (`is_secret=true` rows):** rotation is a single
  `UPDATE app_settings SET value=$1 WHERE key=$2` + the existing
  `reload_site_config` job picks up the new value within 60s.
  `poindexter settings set <key> <value>` already does this. **Action:**
  add a thin `poindexter auth rotate <key>` wrapper that prompts for
  the new value, validates against an optional "test the new
  credential" hook (e.g. for Discord webhook, POST a heartbeat ping),
  and confirms reseat. The test hook is the rotation-safety lever —
  operator gets a quick "yes, the new webhook works" signal before
  assuming the old one is dead.

- **OAuth `client_secret_encrypted`:** add `poindexter auth rotate-client
<client_id>` — generates a new secret, updates `oauth_clients`,
  invalidates the old one. Currently no rotate command exists.

- **Master `POINDEXTER_SECRET_KEY`:** this is the hard one. Re-encrypting
  every `is_secret=true` row is a write-amplifying operation. Wrap
  the existing `plugins.secrets.rotate_key()` helper in a CLI command
  `poindexter auth rotate-master-key`, gate behind a confirmation
  prompt + a maintenance-mode flag that freezes new writes during
  the rotation window. Document the procedure (downtime cost: ~30s
  for ~50 rows).

- **LIVEKIT_API_KEY / LIVEKIT_API_SECRET:** currently env-only — no
  rotation surface. **Action:** move to `app_settings` `is_secret=true`,
  read via `get_secret` at JWT-mint time (the `/voice/join` route is
  the only reader). Then they rotate like every other DB-backed secret.

- **Cloud DB DSN:** also env-only. Migrate to bootstrap chain so
  rotation is a bootstrap.toml edit + worker restart (no full
  redeploy needed).

- **Bootstrap.toml secrets (DB password, Telegram bot token):**
  these bootstrap the system before app_settings is reachable, so
  they can't move to DB. Document the rotate procedure: stop worker,
  edit bootstrap.toml, restart, verify.

### Long-lived-secret watchdog

- New column on `oauth_clients`: `last_rotated_at TIMESTAMPTZ DEFAULT NOW()`.
- New shadow table `secret_metadata` keyed on `app_settings.key` with
  the same `last_rotated_at`.
- New job `services/jobs/secret_rotation_watchdog.py`, schedule = every
  24h. For each tracked secret, if
  `now - last_rotated_at > rotation_interval_days` (default 90), emit
  an `r2_static_drift`-style operator finding via
  `utils.findings.emit_finding`. Per-secret override on the rotation
  interval.

## What's done right (don't break these)

- OAuth code path uses `hmac.compare_digest` for client_secret compare
  (timing-safe).
- Auth code TTL = 5 min; JWT TTL = 60 min (both conservative).
- `plugins/secrets.py` envelope encryption (pgcrypto + `enc:v1:`
  sentinel + version prefix) is clean and rotation-friendly.
- Brain daemon refuses to start when `POINDEXTER_SECRET_KEY` missing —
  fail-loud, matches the no-silent-defaults rule.
- Lemon Squeezy webhook HMAC verification uses the async
  `get_secret` path correctly.
- All workflows use `pull_request`, not `pull_request_target` — PR
  code never gets the privileged token.
- `aquasecurity/trivy-action` and `release-please-action` are SHA-pinned.
- `package-lock.json` has 0 missing integrity hashes and 0 install
  scripts on any dependency.
- Production code is free of `eval` and `exec` on request-time inputs;
  the lone unsafe-deserialization branch (memory_system fallback)
  reads only from our own DB.
- SQL identifier interpolation paths go through the allowlist
  validator or hardcoded literals; values themselves use `$N` params.
- The 2026-05-09 LiteLLM proxy CVEs (SQLi/SSTI/RCE) were patched
  proactively in commit `cc8e4c5b`. We're past every known LiteLLM
  attack version.

**CI guard:** `scripts/ci/check-action-pins.py` walks
`.github/workflows/` and fails the build if any `uses:` value isn't

trailing `# <tag>` comment so the audit trail stays intact.

### P1 #16 — Workflow `permissions:` tightened (2026-05-12)

`unit-tests.yml`, `migrations-smoke.yml`, and `grafana-panels-lint.yml`
each gained a top-level `permissions: { contents: read }` block. The
other workflows already had explicit `permissions:` declared at
workflow root (`security.yml`, `codeql.yml`) or per-job
(`sync-to-public-poindexter.yml`). `release-please.yml` and
`release-poindexter-to-pypi.yml` keep their elevated permissions
(`contents: write` + `pull-requests: write` and `id-token: write`
respectively) because those jobs legitimately need them.

This closes the `GITHUB_TOKEN`-defaults-to-write-all gap — any
future privilege requirement now has to be explicitly declared
where it's needed, not silently inherited.
