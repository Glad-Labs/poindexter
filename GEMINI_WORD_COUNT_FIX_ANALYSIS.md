# Blog Post Word Count Bug Fix - Investigation Report

## Problem Summary

**Issue:** 37% of published blog posts (14 out of 38) have insufficient word count
- 14 posts under 1000 words (37%)
- Target is 1500 words (with ±30% tolerance = 1050-1950 words acceptable)
- Shortest post: 380 words
- Average post length: 1428 words (below target)

## Root Cause Found

The **Gemini API code path** in `ai_content_generator.py` was bypassing word count validation and refinement logic.

### Technical Details

**File:** `src/cofounder_agent/services/ai_content_generator.py`

**Gemini Path (Before Fix)** - Lines 507-528:
```python
if generated_content and len(generated_content) > 100:
    validation = self._validate_content(generated_content, topic, target_length)
    # ... metrics setup ...
    return generated_content, metrics["model_used"], metrics  # ← RETURNS IMMEDIATELY
```

**Ollama Path (Correct Implementation)** - Lines 659+:
```python
if validation.is_valid:
    return generated_content  # Only return if valid

if metrics["refinement_attempts"] < self.max_refinement_attempts:
    # Perform refinement with constraints
    refined_validation = self._validate_content(...)
    if refined_validation.is_valid:
        return refined_content  # Return only after passing validation
```

### Why This Happened

1. **Gemini was the default provider** (line 240 in ai_content_generator.py)
   - Used when API key is configured and no provider specified
   - Selected first in the provider fallback chain

2. **Validation logic present but unused** (line 508)
   - Validation runs and scores content
   - But return statement (line 528) ignores the validation result
   - Content returned regardless of whether `validation.is_valid` is True/False

3. **No refinement attempts**
   - Gemini path had no retry logic
   - If content failed word count check, it was accepted anyway
   - Ollama path (lines 676-762) attempts 3 refinements before accepting failure

## Solution Implemented

**File Modified:** `src/cofounder_agent/services/ai_content_generator.py`

**Changes:**
1. After Gemini generates content and validation runs (line 508)
2. Check if validation.is_valid (line 528) - only return if True
3. If validation fails AND refinement attempts remaining (line 541)
4. Attempt refinement with word count constraints  (lines 547-624)
5. Validate refined content and return if valid (line 607)
6. If refinement exhausted, return best attempt with warning (lines 626-641)

**Code Pattern Added:**
```python
# If content passes, return immediately
if validation.is_valid:
    return generated_content

# If content fails but refinement available, retry
if metrics["refinement_attempts"] < self.max_refinement_attempts:
    metrics["refinement_attempts"] += 1
    refinement_prompt = get_refinement_prompt(...)
    refined_validation = self._validate_content(...)
    if refined_validation.is_valid:
        return refined_content

# If all refinements exhausted, return best attempt with warning
return generated_content
```

## Word Count Validation Rules

**Target Length Source:** Frontend sends `word_count` → Backend translates to `target_length`

**Validation Logic** (ai_content_generator.py line 118-126):
```python
word_count = len(content.split())
min_words = int(target_length * 0.7)      # 70% of target
max_words = int(target_length * 1.3)      # 130% of target

if word_count < min_words:
    issues.append(f"Content too short: {word_count} words (target: {target_length})")
    score -= 2.0
```

**Example with target=1500:**
- Acceptable range: 1050-1950 words
- Under 1050 words: Validation fails, triggers refinement
- Over 1950 words: Validation fails, triggers refinement
- After 3 refinement attempts: Returns with warning

**Quality Threshold:** `self.quality_threshold` (default: 7.0/10)

## Impact Analysis

### Before Fix

- Refinement was never attempted

### After Fix

- Automatic refinement attempts when content is short
- Better content quality through iterative improvement
- Consistent target length across all generated posts
