# Video Composition — Director / Sources / Renderer

**Status:** Design proposal (2026-05-28). Captures the architecture for moving from
single-source video generation ("always Ken Burns" OR "always Wan2.1") to a
director-driven, multi-source approach where each post gets a custom shot list
mixing image-gen stills, Pexels stock footage, Wan2.1 short clips, and Ken-Burns
treatments — chosen per-shot by an LLM director, rendered by the existing
`MediaCompositor` plugin.

This doc is the contract. The implementation lands in 3 PRs (sequenced at the
bottom). Each PR is independently shippable and useful.

## Background — what's broken today

`services/video_service.py` hardcodes a single composition strategy: generate
8 image-gen stills, stitch them with Ken Burns motion + the podcast audio, hand off
to a video server. There are no per-post creative decisions.

At the same time, the wan-server was migrated to Wan2.1 text-to-video, but
`video_service.py` was never updated to send the new schema — so every video
generation has been failing with HTTP 422 (the wan-server expects a `prompt`
field; the client sends `image_paths` + `audio_path`).

Both strategies on their own have produced unimpressive results so far:

- **Ken Burns slideshow** — feels static; doesn't match "video" expectations
- **Wan2.1 native T2V** — low-fidelity at 1.3B params; uncanny-valley artifacts

The right answer is neither — it's **mixing them per-shot based on what each
shot needs**.

## Goal

Each video gets a **shot list** — an ordered sequence of shots, each with a
target duration, an intent, a content source (image-gen / Pexels / Wan2.1 / Ken
Burns / future), and a generation prompt or stock query. The compositor
fetches the per-shot media, assembles into the final MP4, overlays the
narration audio.

**Decisions get made in three layers**:

1. **The director (LLM call)** — reads the post + podcast script, outputs a
   shot list as JSON. Initially heuristic prompting; later trained by the
   operator's approve/reject feedback.

2. **The source plugins** — each shot's `source` value resolves to an
   existing plugin (already have image-gen, Pexels, Wan2.1 video provider) or a
   new one (Ken Burns wrapper over image-gen).

3. **The renderer (`MediaCompositor`)** — already exists as a protocol with
   an `ffmpeg_local` implementation. Takes a list of `CompositionScene`s
   (clip + narration + duration) and produces an MP4.

The seam between layers is the **shot list schema** — auditable, testable,
regeneratable independently of the renderer.

## Shot list schema

Stored as JSONB on `posts.video_shot_list`. Generated by the director stage
during the pipeline, BEFORE the post hits `awaiting_approval`. The operator
can review the shot list as part of post review (alongside content) — fixing
problems at the shot-list level is cheaper than re-rendering.

```json
{
  "version": 1,
  "total_duration_s": 47,
  "shots": [
    {
      "idx": 0,
      "duration_s": 6,
      "intent": "establish topic — set the scene",
      "source": "pexels",
      "query": "data center server room lights",
      "narration_offset_s": 0
    },
    {
      "idx": 1,
      "duration_s": 5,
      "intent": "abstract — illustrate the metaphor",
      "source": "image_kenburns",
      "prompt": "an open glass door with abstract data flowing through, cinematic lighting",
      "kenburns_zoom": [1.0, 1.2],
      "narration_offset_s": 6
    },
    {
      "idx": 2,
      "duration_s": 4,
      "intent": "concrete — Wan2.1 motion clip for variety",
      "source": "wan21",
      "prompt": "a hand opening a transparent glass door in slow motion, depth of field",
      "narration_offset_s": 11
    }
  ],
  "director_model": "glm-4.7-5090",
  "director_prompt_version": "v1",
  "director_decided_at": "2026-05-28T00:30:00Z"
}
```

**Field semantics**:

- `idx` — zero-based shot index; the renderer concats in this order
- `duration_s` — target wall-clock duration of this shot in the final video
- `intent` — natural-language note from the director ("why this shot exists").
  Useful for debugging + the quality feedback loop.
- `source` — discriminator for which plugin renders this shot. Initial set:
  `"image_gen"` (static still), `"image_kenburns"` (still + motion), `"pexels"`
  (stock clip), `"wan21"` (native T2V), `"holdover"` (fade from previous shot
  — used for breathing room).
- `prompt` / `query` — generation prompt OR stock-library query, depending on
  source. The director chooses which is appropriate.
- `narration_offset_s` — where in the podcast audio this shot starts. The
  renderer slices the audio rather than re-narrating per shot. This keeps the
  podcast as the single source of voice and lets the director focus on
  visuals.
- `director_*` — provenance for the feedback loop. The next-gen director can
  read these to learn what its predecessor produced.

## Director stage

A new pipeline stage between `generate_media_scripts` (already exists, gives
us the podcast script) and `finalize_task`. The stage:

1. Loads the post body + podcast script + (optional) niche-level style
   preferences.
2. Calls the LLM via the existing `dispatch_complete` router with the
   `director_prompt` from UnifiedPromptManager.
3. Parses the JSON output, validates against a Pydantic model, persists to
   `posts.video_shot_list`.

The prompt lives in YAML + Langfuse like every other production prompt
(`feedback_prompts_must_be_db_configurable`). Initial sketch:

> You are the video director for a Glad Labs blog post. Given the post body
> and podcast script below, produce a shot list of 6-12 shots covering the
> full podcast duration. Mix sources for variety:
>
> - **Pexels stock** for concrete real-world subjects (people, places,
>   products, recognizable scenes)
> - **image-gen stills with Ken Burns motion** for abstract concepts and
>   metaphors that need a custom look
> - **Wan2.1 short clips** when a shot benefits from native motion (water,
>   wind, hands manipulating objects) — keep under 6 seconds
> - Never more than 2 consecutive shots from the same source.

Output: validated JSON per the schema above.

## Per-source plugin contract

Each `source` value resolves to one of:

| `source`         | Renderer               | Output            | Notes                                                          |
| ---------------- | ---------------------- | ----------------- | -------------------------------------------------------------- |
| `image_gen`      | existing image-gen svc | PNG → static clip | Held for `duration_s` as still frame                           |
| `image_kenburns` | image-gen + KB wrapper | PNG + Ken Burns   | Today's "Ken Burns slideshow" code, refactored into one plugin |
| `pexels`         | new — Pexels API       | MP4 clip          | Trim/loop to `duration_s`; respect license                     |
| `wan21`          | existing wan-server    | MP4 clip          | New schema: send `prompt`, get `duration_s` clip               |
| `holdover`       | renderer-internal      | fade transition   | No source plugin; renderer cross-fades                         |

Each plugin returns a path to a local file. The renderer treats them all the
same — concat in shot-list order, transition between, overlay narration.

The existing `MediaCompositor` Protocol in
`src/cofounder_agent/plugins/media_compositor.py` (CompositionScene +
CompositionRequest dataclasses) is what we hand the shot list to once each
shot's clip is resolved. `ffmpeg_local` already implements the
local-rendering side.

## Quality feedback loop

The Layer 2 media-quality eval (planned in a separate PR after #648) compares
each generated video to the source post for faithfulness. When the operator
rejects a video, the rejection note + the offending shot list get logged to
`audit_log` with `event_type='media_director_feedback'`. Once we have ~50
labelled examples per niche, train-time augmentation:

- Aggregate which source mixes correlate with low reject rates per niche
- Surface to the director prompt as a "preferred mix" hint
- Eventually fine-tune a director model per niche (the `auto_approve`
  trust-build cycle from `feedback_auto_publish_requires_edit_distance_track_record`)

This is the same pattern as the text auto-approve gate — start with operator
judgment as the gold signal, build trust gradually, automate on track record.

## Decision matrix — what source for what

The director is the SMART decider; the matrix below is the FALLBACK / prior
that the director's prompt encodes initially. As the feedback loop matures,
the matrix is what the director's policy converges toward.

| Shot intent                       | Recommended source                 | Why                                          |
| --------------------------------- | ---------------------------------- | -------------------------------------------- |
| Real-world subject (person/place) | Pexels                             | Stock photography > AI hallucination         |
| Abstract concept / metaphor       | image-gen still + Ken Burns        | Custom aesthetic; AI handles abstraction     |
| Motion / time-evolution           | Wan2.1 short clip                  | Native video > static frame with KB pan      |
| Breathing room / transition       | Holdover with fade                 | Lets the audio breathe; no compute cost      |
| Establishing shot                 | Pexels OR image-gen+KB             | Whichever has stronger visual on first beat  |
| Punchy closer                     | image-gen + KB (high-impact still) | Posters work; native AI motion often fizzles |

**Pacing rules** (in the director prompt):

- Never more than 2 consecutive shots from the same source
- Total run time matches the podcast duration ±1s
- First and last shots are not Wan2.1 (its artifacts are most visible at
  attention peaks)

## Implementation plan — 3 PRs

### PR 1 — shot list schema + director stage (foundation)

**Scope:**

- Migration: add `posts.video_shot_list jsonb` column
- New stage `modules/content/stages/generate_video_shot_list.py` between
  `generate_media_scripts` and `finalize_task`
- Pydantic models for shot list validation (`schemas/video_shot_list.py`)
- Director prompt added to UnifiedPromptManager YAML + Langfuse
- Pipeline template updated to include the new stage
- Tests: shot list schema validation, director stage with mocked LLM

**Out of scope** (deferred to PR 2):

- Actually using the shot list for rendering — the existing pipeline keeps
  running with the hardcoded approach. The shot list lands in the DB but
  doesn't drive anything yet. This is intentional — we want to see the
  director's output for a few real posts before committing the renderer to it.

**Estimated effort:** 2-3 focused hours

### PR 2 — per-source plugins + renderer wiring

**Scope:**

- New `pexels_video.py` plugin (stock-clip lookup via Pexels API)
- Refactor existing image-gen + Ken Burns logic from `video_service.py` into a
  `image_kenburns` source plugin
- Fix wan-server schema mismatch — update `video_service.py` to send `prompt`
  to `/generate` (or wrap the wan-server call in a `wan21` source plugin
  with the correct request body)
- New `services/video_director_service.py` — orchestrator that reads
  `posts.video_shot_list`, calls per-source plugins, hands result to
  `ffmpeg_local` compositor
- Replace the current monolithic `generate_video_for_post` in
  `video_service.py` with a thin wrapper that delegates to the new
  orchestrator
- Tests: per-source plugin contract tests, end-to-end orchestrator with
  mocked plugins

**Estimated effort:** 4-6 focused hours

### PR 3 — quality feedback loop

**Scope:**

- Layer 2 media quality eval (Whisper transcript → LLM faithfulness scoring
  vs source post) — was already planned as the follow-up to #648
- Director rejection logging — when operator rejects a video via
  `poindexter media reject`, log the rejected shot list + rejection note
- Director prompt picks up rejection signal in its preamble (read trailing-N
  rejections for the niche, surface as "avoid this kind of mix")
- Grafana panel showing per-source approval rates per niche

**Estimated effort:** 3-4 focused hours

## Open questions

1. **Variants per post** — do we want a single video per post, or short + long
   variants (the existing `video_short` medium)? Current design assumes one
   shot list per post; `video_short` would need its own director call with
   different pacing prompt. Probably fine — different `media_to_generate`
   values fan out to different shot lists.

2. **Operator review surface for the shot list** — does Matt want to see the
   shot list at approval time (alongside post content), or only review the
   rendered video? Showing the shot list lets him fix problems before the
   ~5min render runs. Defer to PR 1 to land before deciding.

3. **Director model** — ~~pin to one model or use the cost-tier router?~~
   **Resolved (#1712):** pinned to the writer model via the `video_director_model`
   setting (defaults to `ollama/gemma-4-31B-it-qat:latest`, same model as
   `pipeline_writer_model`). The director's structured-JSON output is well-served
   by the same writer-grade capability. `glm-4.7-5090` was the writer at design
   time; it is now the reviser (`qa_rewrite_model`) since the 2026-06-18 bakeoff
   (#1692).
